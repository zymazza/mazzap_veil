# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

VEIL is a standalone, region-agnostic, fully georeferenced 3D digital-twin engine: point it at a DEM (and optionally imagery and map layers) and it builds a browser-viewable twin of that ground. No database server, no cloud, no build step at view time: a zero-dependency Node static server (`server.js`) serves a Three.js viewer (`public/`) over a self-contained bundle of geospatial data (a twin's `data/` or any `--data-dir`/`TWIN_DATA_DIR`). Nothing is fetched from the network at view time. **Twin data is private and gitignored** — the repo ships only code plus one small demo AOI (`packs/us-national/demo/`); see "Privacy" in the README.

The system of record is the **twin store** — canonically, the append-only write journal in `<data>/journal/`, materialized as a GeoPackage at `<data>/twin.gpkg` (see "Twin store" below). The flat JSON the viewer loads is an **export** of the store — the viewer is a window on the store, not the product.

## Commands

```bash
npm start                 # serve at http://127.0.0.1:4173 (PORT/HOST/TWIN_DATA_DIR env override)
npm run demo              # build the Flatirons demo twin into twins/demo/data (needs internet + GDAL)
npm run serve-demo        # serve the demo twin at http://127.0.0.1:4174
npm test                  # query-layer tests against the offline committed fixture twin
npm run test:demo         # query-layer tests against the real-data demo twin (builds it if missing)
npm run build-test-fixture # regenerate the committed offline test fixture twin
npm run build-from-aoi    # build a whole US twin from one AOI polygon (3DEP + NAIP + LANDFIRE)
npm run ingest-dem        # genesis: any DEM (+ AOI) -> terrain grid + georef.json
npm run ingest-imagery    # align aerial imagery to the terrain footprint
npm run add-layer         # import any vector/raster file as a draped, clickable layer
npm run rebuild-store     # reconstruct <data>/twin.gpkg from its journal
npm run migrate           # genesis only: seed a store (+ journal) from a flat data/ bundle
npm run export            # regenerate every viewer JSON payload from the store
npm run build-vegetation  # tree/shrub instances -> store -> export (analyze_vegetation.py + build_surrounding_vegetation.py)
npm run canopy-history    # temporal query: canopy density per pipeline run (--bbox to scope)
npm run build-survey-package  # generate the QField survey package (Survey companion)
python3 scripts/mcp_server.py  # MCP server (stdio) over the twin store — see docs/mcp.md

docker compose up --build              # serve in a container (Node + full Python pipeline, pinned)
docker compose run --rm veil <cmd>     # run any npm/pipeline step in that image (twins persist to ./twins)
```

`npm test` runs `scripts/twin_query_test.py`, which asserts against the deterministic Flatirons demo twin (`twins/demo/data`, built from the committed AOI on first run; set `TWIN_DATA_DIR` to test another twin). Visual verification: `scripts/screenshot.js` is a headless Playwright harness (`node scripts/screenshot.js <out.png> [layerLabelsCsv] [waitMs] [click]`) that loads the viewer against a running `npm start`, optionally isolates layers by label, clicks the canvas to exercise click-to-identify, and reports page/console errors (`npm i -D playwright` first; it is not a dependency).

The Python pipeline scripts need GDAL (`osgeo`), numpy, pyproj, and Pillow; `requirements.txt` pins the pip-installable subset (`mcp`, `pyproj`, `Pillow`) and documents that GDAL/numpy come from the OS, not pip (the bindings must match the system GDAL, and `ReadAsArray`/`WriteArray` need a matching numpy). The fetch scripts (`national_fetch.py`, `packs/us-national/fetch_landfire.py`, `build_from_aoi.py`) need internet for one-time API snapshots. The build scripts are one-time/occasional data-pipeline steps; day-to-day viewer work needs only `npm start`.

**Containerized path** (`Dockerfile`, `docker-compose.yml`, `.devcontainer/`): the whole toolchain pinned and pre-built. The image is based on the official OSGeo GDAL `ubuntu-full` image (ships python3 + version-matched `osgeo` bindings + numpy + array support — the part that's painful to build), with Node (NodeSource, `NODE_MAJOR`) and the pip deps layered on; GDAL is pinned via the `GDAL_VERSION` build arg. server.js binds `HOST=0.0.0.0` in the container. `docker-compose.yml` bind-mounts `./data` and `./twins` so pipeline output persists to the host (and stays gitignored); `docker compose run --rm veil <cmd>` runs any pipeline step in the same image. The Dev Container reuses the Dockerfile for an editable environment. Local `npm`/`python3` use is unchanged — the container is purely an adoption convenience.

## Twin store

The canonical artifact is the **write journal**: `<data>/journal/NNNNNN-<script>.jsonl.gz`, append-only at the file level (one file per write session; existing files are never modified). The journal lives inside the twin's data dir and is private along with it. Every store write is journaled, and `npm run rebuild-store` reconstructs `twin.gpkg` from the journal exactly — same runs, same timestamps, same observation order (verified row-identical). The gpkg itself is a **materialized index**: fast to query, safe to delete, rebuilt from the journal. (If you carry a twin's `data/` to another machine, `npm run rebuild-store` regenerates the gpkg from its journal.) `npm run migrate` is genesis only — it seeds a brand-new store (journal file 000001) from a flat bundle; on an existing store it is an idempotent no-op. If a run crashes mid-session the gpkg can be ahead of the journal (the journal flushes on close); `rebuild-store` restores the journaled truth.

`twin.gpkg` is one GeoPackage holding both the spatial layers and the relational tables. **All Python access goes through `scripts/twin_store.py`** — writes via `begin_run()`, `upsert_entity()`, `observe()`, `bulk_upsert_vegetation()`, `reconcile_membership()`; reads via `latest()`, `history()`, `latest_attrs()` and **`instances()`**, the read API that returns a whole population (alive + member, ordered by entity_id). Build scripts consume `instances()`, never the exported JSON — the exports are pure derivations for the viewer.

- **Tables:** `meta` (origin_utm, crs, schema_version, scene template, vegetation metadata), `pipeline_runs` (one row per build-script execution), `entities` (deterministic ID, kind, created/retired run), `observations` (append-only attribute history; current state = latest row per entity+attr), `layers` (atlas layers **and** registered input files — terrain grids, imagery, a LiDAR seed — each with `content_sha1`, so "did an input change" is queryable). **Spatial layers:** `trees`/`shrubs` (points with `entity_id`, `x`, `y`), `parcels`, `streams`, `roads`, `building_footprints`, plus `atlas_<layer_id>` for each localized atlas vector and `survey_<layer>` for survey features. Rasters and model binaries stay as files on disk, registered by hash.
- **Coordinates are scene-local meters** (x = east, y = north; the twin's projected CRS minus `origin_utm`) — the one convention, recorded in `meta`. Spatial layers use gpkg srs_id −1 (undefined cartesian).
- **Entity IDs are deterministic:** `"<kind>:" + sha1(f"{source}|{round(x,1)}|{round(y,1)}")[:12]` from the persisted (3-decimal) coordinates; named entities (`building_model:<id>`) use natural keys. Rare same-rounded-position collisions get a deterministic `-2`/`-3` suffix resolved by position probing, so a given physical stem keeps its ID across rebuilds. Both vegetation RNGs are seeded, so rebuilds with unchanged inputs are exact no-ops against the store (zero new entities/observations). **Identity is positional by design:** a stem that *moves* (only the clearance pass moves stems) becomes a new entity and the old one is retired — lineage across moves is intentionally severed, not tracked.
- **No deletes:** an entity that disappears from a rebuild is *retired* (`retired_run_id`/`retired_at`); one that reappears is un-retired. A tree can belong to both the AOI and surrounding payloads — one entity with `member_parcel`/`member_surrounding` observations; each build script reconciles only its own membership flag.
- **Flow:** build script → store (`bulk_upsert_vegetation` + `reconcile_membership`) → `export_viewer_payloads.py` rewrites the viewer JSON (each tree/shrub carries its `id`; the viewer ignores it). Never hand-edit the exported JSON — the next export overwrites it, and nothing reads it but the viewer. If a twin has pristine LiDAR stems (`<data>/vegetation/tree_instances.lidar.json`), that file is an immutable input — the seed `analyze_vegetation.py` regenerates from; its file order also pins the RNG sequence. Temporal queries are plain SQL over `entities`/`observations`/`pipeline_runs`; `scripts/canopy_density.py` (`npm run canopy-history`) is the reference consumer (per-run liveness + latest-attr-as-of-run pattern, optional `--bbox`).
- **Building placements:** the viewer editor POSTs to `/api/building-placements`; `server.js` updates `manifest.json` (unchanged viewer flow) *and* appends `{ts, placements}` to `<data>/buildings/models/placements.log.jsonl` (the Node→Python handoff; the server never touches the gpkg). `scripts/ingest_placements.py` (called automatically by the exporter) turns new log lines into timestamped `placement` observations, at which point they also enter the journal. `manifest.json` is the one file with two writers (server on save, exporter from the store); they converge once ingest runs.
- `clear_vegetation_around_buildings.py` is a store-to-store post-pass: it reads the AOI population via `instances()`, pushes/deletes stems around the placed models, and writes the cleared population back (moved stems become new entities — IDs are position-derived — and old positions leave `member_parcel`; on re-clearing, previously retired cleared positions reactivate). Run it after `npm run build-vegetation`, which regenerates the uncleared population.

## MCP server

`scripts/mcp_server.py` (Python, official `mcp` SDK from `requirements.txt`, stdio) is the agent-facing query surface over the store — the store is **read-only**; the only write tools are the map drawings (below), which never touch the store. All logic lives in `scripts/twin_query.py`, tested by `scripts/twin_query_test.py`; the MCP tools are thin wrappers. Register with `claude mcp add veil-twin -- python3 <repo>/scripts/mcp_server.py`. Conventions: points in `{lat,lon}` or scene-local `{x,y}` (outputs echo both; the pyproj transform agrees with `viewer/georef.js` to ~0.1 mm); one `region` abstraction with four shapes (`aoi` / `bbox` / `within_m`+`point` / `polygon`, resolved solely by `twin_query.resolve_region`); provenance (`source`/`confidence`/`run_id`/`observed_at`, or layer `acquisition`/`service`) on every fact; structured `{"error": …}` payloads. The point-identify tools are a port of app.js click-to-identify (same point-in-polygon / 8 m line-hit / grid-legend / species-grid-bitmask logic); `canopy_change` is the canopy_density.py query as a tool, region-scopable. Tool list and example sessions: `docs/mcp.md`.

**Survey companion on MCP:** the `survey_*` kinds are first-class for `find_entities`/`summarize_region`/`aggregate_entities`; `list_survey_layers` is the discovery tool (uploaded layers, kinds, fields, photo presence), and `identify_at` now includes survey features (photo + status) — closing the old atlas-only gap. All read straight from `<data>/surveys/` (no `server.js` involvement). (This engine has no hydrology/simulation surface — that lives in the property-specific `snow-road-twin` checkout.)

**Map drawings:** `draw_polygon` / `draw_point` / `clear_drawings` let the LLM put labeled **orange** shapes on the live 3D map instead of reciting coordinates — they append to `<data>/annotations.json` (scene-local meters, atomic rewrite; never the store/journal), which `public/annotations.js` polls every 4 s and renders as terrain-hugging outlines, markers, and label sprites. Works identically for the built-in chat and any external MCP client on the same twin. The chat panel's "Clear drawings" button posts to `/api/annotations/clear` (`server.js` empties only the drawings, preserving layer views); chat.js also refreshes annotations as soon as a reply lands.

**Layer views:** the same channel lets the LLM drive the twin's *own* atlas layers, written into `annotations.json` under a `layer_views` list (validated by `twin_query.py` against the atlas catalog; never the store). `set_layer_visibility(layer_id, visible)` toggles a layer's drape; `filter_layer(layer_id, values, field?)` turns a layer on but reveals **only** the selected regions — legend class names for categorical rasters (re-rendered from the value grid in legend colors), modeled-habitat species for the GAP richness grid (an orange habitat mask from `gap_species_grids`, e.g. "where could I find wild turkey"), or `__label`/attribute values for vectors. `reset_layer_views` drops every override. `public/annotations.js` hands the `layer_views` array to `public/app.js` (which owns the drape) via `window.__twin.applyLayerViews`; it's **edge-triggered** — applied on file change, with manual user toggles winning in between and reclaiming a layer from the agent. `filter_layer` reports `matched_values`/`unmatched_values` (case-insensitive), and `layer_summary` exposes the filterable class/label/`filterable_species` values for discovery.

**Chat panel ("Ask the land"):** the viewer's collapsible chat window (`public/chat.js`, stacked above the coordinate readout) talks to `POST /api/chat` in `server.js`, which spawns `scripts/mcp_server.py` once and bridges its tool catalog to an LLM's function calling. **Two providers, one tool loop** (`CHAT_PROVIDER`, default `openai`; inferred `ollama` if `OLLAMA_MODEL` is set): `openai` drives **gpt-5.5** over the Responses API (`previous_response_id` chaining; chat/completions rejects tools+reasoning for this model); `ollama` drives a **local model** over Ollama's native `/api/chat` (`OLLAMA_HOST` default `http://127.0.0.1:11434`, `OLLAMA_MODEL` default `gpt-oss:20b`, `OLLAMA_NUM_CTX` default 98304 — the native endpoint, unlike Ollama's OpenAI-compatible `/v1/chat/completions`, honors `num_ctx`, which we need to size the context so a long tool session never truncates (gpt-oss's KV is cheap: even the full 131072 fits ~17 GB / 100% GPU on a 24 GB card); `OLLAMA_TEMPERATURE` default **0** because gpt-oss defaults to 1.0, which makes tool-calling unreliable — malformed/truncated tool-call JSON or empty answers; `OLLAMA_MAX_TOOL_ROUNDS` default 16 because reasoning models emit **one** tool call per turn, so the OpenAI path's 8-round budget is too tight), replaying the message list across tool rounds — keyless, nothing leaves the machine. `npm run start:local` launches with the Ollama provider. The two paths are `runOpenAI`/`runOllama`, fed a shared neutral `toolDefs` list and each shaping it into its provider's tool schema. `GET /api/chat/config` reports `{provider, model, needs_key}` so the viewer hides the OpenAI "Key" button on a local model. Still zero npm dependencies (built-in `fetch` + `child_process`). Scopes: whole twin; a polygon drawn on the terrain (3+ clicks via "Draw region", clearable — server pre-loads `summarize_region` for it); or a picked point ("Pick point", same terrain raycast as the GPS readout — server pre-loads `identify_at` + `summarize_region` within 100 m). While a draw/pick mode is active the readout's click handler stands down (it checks `__twin.chat.state.mode`). API key, in priority order: a **per-request key** the viewer sends as the `X-OpenAI-Key` header (bring-your-own-key — the "Key" button in the chat panel stores it in the browser's `localStorage` only, never on the server or in the repo), else the server's `OPENAI_API_KEY` env / gitignored `.openai_key` file. Set `OPENAI_REQUIRE_USER_KEY=1` to disable the server-key fallback entirely, so every request **must** bring its own key — use this for any public deployment. Model/effort via `OPENAI_MODEL` / `OPENAI_REASONING_EFFORT` (default `gpt-5.5` / `low`). **The endpoint is otherwise unauthenticated** — without `OPENAI_REQUIRE_USER_KEY`, anyone who can reach the port can spend on the server's configured key (see "Security posture" in the README).

## Survey companion

The QField **write path** into the store (full doc: `docs/survey.md`). `npm run build-survey-package` generates a QField project (`<data>/surveys/package/`: generated `project.qgs` + empty `survey.gpkg` with four fixed layers — `trails`, `stream_centerlines`, `photo_points`, `observations` — + imagery basemap), downloadable from the viewer's "Survey companion" panel. The package is fully derived from the twin: CRS/proj4 from `georef.json`, extent from the terrain grid, basemap from the twin's imagery, title from `scene.json`. After fieldwork the zipped project folder is uploaded back (`POST /api/survey-upload`, raw zip body; optional `X-Survey-Token` against gitignored `.survey_token`). The server is durable-drop-first: zip → `<data>/surveys/incoming/` + line in `uploads.log.jsonl` **before** spawning `scripts/ingest_survey.py --pending` (cursor in meta, like the placements log; `npm run export` is the deferred fallback when the Python side is down).

Ingest rules (don't weaken them): identity is the **natural key** `survey_<layer>:<uuid>` from the form's hidden auto-uuid — never positional; a feature missing from an upload means nothing (no auto-retire — retirement is the explicit `status` field, active/retired/removed); ingest **diffs before writing**, so an unchanged re-upload journals only its `run`/`finish_run` pair (an already-retired feature still on the phone is skipped entirely, no unretire/retire churn); geometry changes write the scene-local GeoJSON as a `geometry` observation (history is a query, not journal archaeology) plus `upsert_feature` into the `survey_<layer>` spatial table (same entity, new shape); `captured_at` is normalized to UTC (naive = device-local assumption); GPS accuracy lives in its own `accuracy_m` attr, **never** the 0–1 `confidence` column; photos follow the model_file pattern (`<data>/surveys/photos/<sha1[:12]>-<name>`, observed as `{path, bytes, sha1, captured_at}`). Exports: `<data>/surveys/<layer>.geojson` + `survey-layers.json` (a catalog separate from the atlas one, which `build_viewer_layers.py` rewrites wholesale); the viewer drapes them, identifies them (points included, photo shown in the card), and `public/survey.js` is the upload panel. MCP sees `survey_*` kinds automatically (find_entities/summarize_region/aggregate_entities), `list_survey_layers` is the discovery tool, and `identify_at` now includes survey features (photo + status) — the old atlas-only gap is closed.

## Engine vs. regional pack

The repo is a **region-agnostic engine** (`scripts/`, `public/`, `server.js`) plus optional **content packs** (`packs/<name>/`). Nothing in the engine names a CRS, a layer, or a species; all of that is pack-supplied or derived from data.

- **CRS is data, not code.** `<data>/georef.json` carries `analysis_crs` (projected EPSG), a `proj4` string, `geographic_crs`, and `origin_utm`. Python reads it through `scripts/twin_georef.py` (`crs()`, `origin()`, `transformers()`, `from_wgs84_transformer()`); the viewer reads it through the vendored **proj4js** (`public/vendor/proj4.js`) in `public/viewer/georef.js`. No module-level CRS constants anywhere. `twin_query.py` derives its lon/lat polygon-detection window from the twin's own extent.
- **Packs** load via `scripts/twin_pack.py`: active pack = `TWIN_PACK` env, else `<data>/pack.txt`, else none. A pack is `packs/<name>/` with `pack.json` and optional hook modules exposing `load(context)`: `vegetation.py` (species/community/type knowledge) and `layers.py` (atlas styles, attribute enrichment, named raster renderings). A pack also owns its own source-acquisition scripts. `packs/us-national/` is the bundled CONUS-wide pack (LANDFIRE EVT typing); it doubles as the reference for the pack interface.
- **Genesis path** (`scripts/ingest_dem.py`): a DEM GeoTIFF + an AOI (bbox or polygon GeoJSON) → `<data>/terrain/grid.json` + `georef.json` + `aoi_local.geojson`, conforming to the **frozen grid contract** (`docs/grid-contract.md` — pin any change there, never edit the interface silently). Working CRS = the DEM's (if projected) or the AOI-centroid UTM zone; origin = center of the outer footprint. `scripts/ingest_imagery.py` aligns aerial imagery to the grid's outer footprint at integer px/m (the invariant `analyze_vegetation.py` depends on). `scripts/add_layer.py` ingests any GeoTIFF/GeoJSON (any CRS) → reproject to scene-local → clip → auto-style → append to `viewer-layers.json` + register in the store. `scripts/build_from_aoi.py` chains all of it from one AOI polygon using live national services (US only).
- **Vegetation capability ladder** (`scripts/analyze_vegetation.py`): LiDAR segmentation stems → DSM−DTM canopy-height-model local maxima (`scripts/veg_detect.py`) → NDVI canopy local maxima → skip (recorded in metadata, not an error). Typing/species come from the pack; with no pack or no NIR, trees are `type:"unknown"` with no species. Deterministic RNG: re-running with unchanged inputs is a 0-observation no-op against the store.

## Architecture

**No modules, no build.** All frontend code is plain scripts loaded via `<script>` tags in `public/index.html` (order matters) that attach globals to `window`: `THREE` (vendored in `public/vendor/`), `VEILViewer` (scene.js), `VEILTerrain`, `VEILVegetation`, `VEILOverlays`, `VEILBuildings3D`, `VEILBuildingEditor`, `VEILGeoref`.

**Boot flow:** `public/app.js` is the application. It fetches `data/scene.json` (the static scene payload: terrain grid, imagery refs, vegetation, vector layers), instantiates the viewer library (`public/viewer/scene.js` + camera/terrain/vegetation/overlays), then layers the twin features on top: atlas layer toggles, the terrain **drape** (atlas layers are rendered into a shared canvas texture that colors terrain pixels, so they conform to topography — never floating geometry), click-to-identify, and the GPS readout.

**Georeferencing:** the scene is in local meters offset from a projected-CRS origin (`origin_utm` in `<data>/georef.json` / `scene.json`). World axes: `x = easting − origin`, `z = −(northing − origin)` (north is −z), `y = elevation − grid.minElevation`. `public/viewer/georef.js` (proj4js, fed the twin's proj4 string) converts to lat/lon in-browser (validated against pyproj). Every script and viewer feature that places things in the scene goes through this convention.

**Atlas data:** every layer in `<data>/atlas/` is a local file indexed in `atlas-manifest.json`, written by a pack's acquisition scripts and/or `add_layer.py` (each layer is tagged with its `acquisition`, e.g. `local_source_clip | api_snapshot`). Layers that legitimately return nothing for a twin's footprint are recorded with their status, not dropped. `build_viewer_layers.py` then produces the viewer-ready forms in `<data>/atlas/local/` (localized vectors with friendly labels, colored raster PNGs + value grids, per-species presence grids) consumed by app.js for draping and identify.

**Vegetation:** `analyze_vegetation.py` detects stems via the capability ladder, types them with the active pack, and plants synthetic `source: canopy_fill` stems to match the imagery canopy. `build_surrounding_vegetation.py` does the same for the terrain apron outside the AOI. Both write to the twin store and re-export the viewer payloads (`tree_instances.json` etc. are exports — regenerable, never hand-edited). The viewer renders AOI and surrounding vegetation as separate toggleable layers using low-LOD tree forms.

**Buildings:** photogrammetry GLBs (private, per twin) are repacked by `build_building_models.py` from `buildings/assets/<id>/` (discovered by folder; metadata supplies the name and source CRS) into `<data>/buildings/models/`, placed per `manifest.json`. The repack registers each run in the twin store: `building_model:*` entities carry the manifest attributes plus a `model_file` observation (path, bytes, sha1, source asset) — the binaries stay on disk, registered like rasters. Pressing **B** in the viewer opens the drag-gizmo editor (`building-editor.js`, three.js TransformControls); "Save Transform" POSTs to `/api/building-placements`, which writes the manifest *and* appends to `placements.log.jsonl` (see "Twin store"). `clear_vegetation_around_buildings.py` pushes/removes vegetation that would pierce the models.

**Surrounding area:** an apron grid (`grid.apron.json`, e.g. from USGS 3DEP) extends terrain beyond the AOI; the viewer's "Surrounding area" toggle renders it with the same surface modes while keeping AOI vegetation inside the AOI.
